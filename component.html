Create a new 'Text and animation' component in the CMS to support HTML5 embeds

1. Screen readers

  • Use ARIA roles and labels:
    • Provide aria-label or aria-labelledby to describe the content.
    • If the animation conveys meaningful information, make sure it's described in text or through aria-describedby.
  • Hide decorative animations:
    • If the animation is purely decorative, use aria-hidden="true" or role="presentation" to prevent it from being announced.
  • Descriptive fallback content:
    • For iframes or SVGs, include alternative text or an accessible summary outside the iframe/SVG.

2. Reading mode (reader view in browsers)

  • Ensure semantic HTML outside of the animation:
    • Reader mode will typically strip non-semantic content, including iframes and canvas-based animations. Ensure the key messages are still conveyed in accessible HTML text nearby.
  • Avoid putting key content inside inaccessible canvases or iframes without a fallback.

3. Keyboard access

  • Make sure iframe content is keyboard navigable:
    • If the embedded content (e.g., interactive SVG or game) has controls or interactivity, test that all elements can be reached and controlled via keyboard (Tab, Enter, Arrow keys, etc.).
  • Focus management:
    • Use JavaScript to manage focus appropriately when users interact with embedded components.

4. ARIA roles and states

  • Role and live regions:
    • Use role="img" with aria-label for non-decorative SVGs.
    • If the animation updates live (e.g., a loading spinner or progress indicator), consider using aria-live="polite" or assertive.
  • Avoid overuse of ARIA:
    • Don’t add ARIA attributes just to “pass” accessibility audits—only use them when they provide meaningful assistance.

5. Graceful degradation/fallbacks

  • No-JS fallback:
    • Provide fallback content for users who have JavaScript disabled or blocked.
  • Reduced motion preference:
    • Respect the prefers-reduced-motion media query. Avoid auto-playing or looping animations if the user has requested reduced motion:
      @media (prefers-reduced-motion: reduce) {
      .animated-element {
      animation: none !important;
      transition: none !important;
      }
      }
  • Timeouts and pause controls:
    • For longer or looping animations, provide controls to pause, stop, or hide the animation (WCAG 2.2 requirement under SC 2.2.2).

6. Content in iframes

  • Title attribute:
    • Always include a descriptive title on iframes, e.g.:
      <iframe src="animation.html" title="Illustration of recycling process" ...</iframe>
  • Accessible iframe content:
    • Ensure the HTML inside the iframe is accessible itself. This is often overlooked.

7. SVG-specific considerations

  • Text inside SVGs:
    • Use <title> and <desc> elements inside the SVG to provide accessible names and descriptions.
  • Role and focusability:
    • Use focusable="false" for purely decorative SVGs.
    • Use tabindex="0" if the SVG is interactive and should be reachable by keyboard.

8. Lottie animations or JSON-based animation libraries

  • Not natively accessible — use them with caution.
  • Provide an accessible alternative (text, static image, or caption).
  • Avoid conveying critical information solely through Lottie unless paired with accessible equivalents.
